# Logic

`internal/ui` now contains:

- `rootModel`, the Bubble Tea application shell
- `queueScreen`, the Musicon-specific queue-management screen
- `playbackScreen`, the Musicon-specific playback screen
- shared UI-facing contracts and small rendering helpers

The root model computes a visually square viewport from terminal size with `components.ClampSquareWithCellWidthRatio`, renders only that centered square during normal operation, keeps queue/playback as dedicated top-level modes, emits periodic Bubble Tea ticks so playback state can refresh while audio is active, and updates the terminal title through Bubble Tea's `View.WindowTitle` field instead of embedding title control sequences in the visible frame content.

Startup now seeds Bubble Tea with an initial terminal size derived from the live stdout TTY when available, then falls back to `COLUMNS`/`LINES`, then `80x24`. `Init` still explicitly requests the live window size so interactive terminals can correct the seeded dimensions immediately. This avoids a blank startup state in PTYs or terminal emulators that never answer Bubble Tea's size probe.

The root model now prefers an explicit configured cell width ratio, still allows `MUSICON_CELL_WIDTH_RATIO` to override that value explicitly, and otherwise falls back to the shared default ratio from `pkg/components`. This keeps square-layout policy stable after the renderer migration while preserving explicit user control.

It now also evaluates explicit minimum viewport requirements before rendering the main shell. If the terminal is smaller than the supported `20×20` minimum, the root model renders a dedicated resize warning and blocks normal application interactions until the terminal is large enough again.

Queue mode arranges source chips, source-specific search-kind chips, optional artist-filter state, search input, and a single merged browser list directly inside the square viewport without adding another persistent panel wrapper. Queued items stay pinned at the top of that browser with a marker, while current search results are appended below them until the user adds them to the queue. The root model now builds Bubble-style key bindings from typed options, passes the queue-specific map into the queue screen, and passes the shared browser-navigation bindings down into the reusable `pkg/components.List`. Queue rows keep normalized provider labels such as `local:`, `radio:`, or `youtube:` visible by passing them as `ListItem.Leading` prefixes instead of baking them into the row title text. That split matters because the shared list's focused-row marquee now scrolls only the trailing title/subtitle region while leaving any leading prefix anchored. Queue focus now follows the visible control stack instead of depending on a required search-toggle shortcut: up/down moves between source chips, search-kind chips, search input, and the merged browser list; left/right cycles the active source or active search kind when one of those chip rows is focused; and pressing up from the first browser row returns focus to search. Printable keys only edit the query while search itself is focused, while source switching, direct search-kind selection, search-kind cycling, collection expansion, removal, and reorder actions stay available from any non-search focus zone. `Ctrl+F` is still accepted as a convenience jump between search and browser, but the user no longer has to rely on it just to reach the query field. The visible chips are now the only search-shaping control: sources such as `all` and `local` expose generic kinds like `All`, `Tracks`, `Streams`, and `Playlists`, while YouTube exposes `Songs`, `Artists`, `Albums`, and `Playlists`. Numeric shortcuts always target the visible chip slots rather than hardcoded semantics, so `1` means "first visible search kind" for the active source, `2` the second, and so on. Live searching still has two phases: a short debounce tick after query changes, followed by a background provider call. Each new query increments a sequence number, cancels any active provider context, and invalidates older debounce or result messages so only the latest request can update the browser. This keeps quit, help, mode switching, and browser movement responsive even when a source is slow or networked. When the active result is a radio stream and the queue is empty, queue mode now still queues the row first but also dispatches one asynchronous playback start command through the injected playback service, so internet radio can begin immediately without blocking the Bubble Tea event loop on stream resolution. The YouTube source now uses that queue model to expose focused songs/artists/albums/playlists modes, let artist rows become a songs-mode filter selection, let album/playlist rows either add the whole collection or expand into child song rows, and synthesize parent queue rows for grouped collection removals while still allowing individual child-track removal. Queue mode also polls the playback snapshot while rendering so the active queued track can be marked inline, and it uses dedicated reorder keys to move queued items while keeping the selected row attached to the moved entry.

Playback mode now renders the selected artwork/lyrics/eq/visualizer pane as the full square background, then composes transport/progress details as a bottom overlay, pane identity as a top overlay, and optional track metadata as an additional top overlay. When album art does not fill the pane, the remaining area now uses a muted filler pattern so the eye can still read the intended artwork viewport. Help no longer swaps to a separate chrome-heavy screen; the root model centers the active screen's help card over the current square body. Playback transport actions are dispatched asynchronously from Bubble Tea's key-handling path and coalesced to one in-flight backend request at a time so rapid key-repeat cannot pin the UI behind slow runtime work such as track resolution. Seek is restored through the same async path, but only as a debounced UI intent: left/right keypresses accumulate a pending adjustment plus deadline, normal playback continues while the user is still tapping, and the next tick after the debounce window issues one absolute seek request against the runtime. The UI deliberately does not render a separate pending-seek indicator beyond the normal status text, so the screen's source of truth remains the last settled playback snapshot until the runtime reports the replacement stream as active. Service-backed playback panes now cache lyrics per normalized track metadata signature and artwork per resolved metadata signature, while visualization panes intentionally re-query their provider on ordinary repaints so live EQ content can advance even when pane and terminal sizes stay fixed. When no artwork or visualization content is currently available and there is no hard error to surface, the playback pane now falls back to a neutral dotted background instead of explanatory placeholder copy, keeping the mode visually calm while preserving overlay controls and pane identity. Lyrics lookup now mirrors artwork lookup in one important respect: the Lyrics pane starts one background request the first time the active track's normalized metadata key changes, keeps the TUI responsive while that request runs, and then renders either `Document.DisplayLines()` output, an instrumental marker, or an explicit empty/error state once the resolver settles. When a lyrics document is longer than the unobscured playback body, the screen keeps a pane-local scroll offset and slices the display lines into a viewport sized to the space between the top pane label/info overlays and the bottom transport overlay. For timed LRC documents, that viewport now follows the active timed row reported by `pkg/lyrics` instead of trusting the manual scroll offset, keeping the current lyric centered when possible and marking it inline for karaoke-style follow-along. Up/down, page up/down, and home/end remain reserved for manual viewport navigation only when the current document is plain or unsynced; in the timed case the screen consumes those keys with a status hint so the active line stays authoritative. Track-to-lyrics normalization stays in `TrackInfo.LyricsRequest()`, which folds basic playback metadata plus the local audio path already carried through cover-art metadata into the reusable lyrics request so local `.lrc` discovery does not need a second parallel field in queue or playback structs. Artwork lookup no longer blocks rendering inside `View`; when the active track or its reusable artwork metadata changes, playback mode starts one background artwork lookup, keeps a centered "recent provider activity" overlay in the artwork pane only while lookup is still running, advances a simple spinner from the root tick, and applies the resulting image once the lookup settles. Recent attempt events are still retained in screen state for debugging and status decisions, but successful artwork loads no longer leave the overlay covering the final image. With a live visualization provider attached, the EQ pane now renders straightforward vertical spectrum bars and the Visualizer pane renders a mirrored spectrum view from the same runtime analysis data, both keeping their hints inside the existing overlay-first playback layout without explanatory placeholders in the empty state.

The package source now also carries package-level and exported-symbol documentation so the root shell, keymap types, and UI contracts remain legible from Go docs in addition to the existing inline layout notes.

Nil or partially configured services are treated as valid UI states: the screens stay navigable and show empty-state messaging rather than inventing missing backend behavior. When playback services are present, repeat and stream toggles route through the injected runtime instead of living only in local screen state, and the root shell uses the latest playback snapshot to keep the terminal title in sync with visible state. Search results and queue entries can now carry reusable artwork metadata so local file paths, embedded-art hints, and external IDs survive queueing before a concrete source resolver exists. Artwork providers then receive normalized reusable cover-art metadata and playback mode delegates protocol detection and rendering to a reusable `pkg/components.TerminalImage` widget. A small adapter in `internal/ui` bridges reusable `pkg/coverart` resolvers to the UI-facing `ArtworkProvider` contract without pushing Bubble Tea or component types into the reusable package.

## Decisions

- Chose dedicated fullscreen queue and playback modes over a persistent split layout because the user explicitly selected that interaction model.
- Chose a square-centered viewport over using the entire terminal canvas because the user explicitly required the app to always fit inside a `1:1` frame.
- Chose an explicit `20×20` minimum terminal policy over allowing arbitrarily small degraded layouts so queue and playback screens fail clearly instead of becoming cramped and misleading.
- Chose interface-only backend integration over stub source or playback implementations because the user explicitly asked for UI-only work at this stage.
- Kept backend construction outside `internal/ui` even after adding a real audio runtime so the UI remains decoupled from concrete playback wiring.
- Chose terminal-title control in the root UI shell over a separate runtime service because the title is presentation state derived from what the UI is already rendering.
- Chose a root-owned help toggle with screen-specific help views so shared chrome stays consistent while each mode can document its own controls.
- Chose to keep terminal-image library usage out of `internal/ui` by pushing rendering into `pkg/components`, so playback mode remains focused on choosing panes and handling fallback messaging instead of protocol details.
- Chose to pass full cover-art metadata into artwork providers instead of a bare track ID so the future reusable provider chain can use local paths, embedded art, and multiple external IDs without making the playback screen aware of lookup policy.
- Chose to pass full reusable lyrics metadata into lyrics providers instead of a bare track ID because strict remote matching and local `.lrc` discovery both need richer context than queue identity alone can provide.
- Chose a pane-local lyrics viewport over expanding the playback layout or reusing queue-mode list chrome because the user wanted long lyrics to stay inside the album-art area and scroll only when needed.
- Chose to seed Bubble Tea with a best-effort initial terminal size instead of waiting only on `RequestWindowSize`, because some PTYs never answer the size query and would otherwise leave the UI stuck in the zero-dimension loading state.
- Chose a shared fixed default cell width ratio over renderer-library auto-detection during the Chafa migration because the user explicitly asked for config-or-fallback behavior once `go-termimg` was removed.
- Chose a single merged queue browser over separate results and queue panes because the user wanted queued items to remain persistent and visually prioritized while search hits continue to appear after them in the same navigation flow.
- Chose explicit queue-reorder shortcuts plus an inline now-playing marker over a separate queue-edit mode because the user wanted the merged browser to stay direct and comfortable without more focus choreography.
- Chose asynchronous queue search dispatch over inline provider calls because the new YouTube source can perform slower yt-dlp-backed work and the user reported that inline searches were freezing input.
- Chose a config-driven keymap layer backed by Bubble bindings over scattering hardcoded `keypress.String()` comparisons through the UI because the user wanted all existing shortcuts to be configurable from TOML.
- Chose a directional focus ladder across sources, modes, search, and browser over requiring a dedicated search-focus toggle because the user wanted arrow-key navigation to reach the query field naturally; kept `Ctrl+F` only as a convenience jump between search and browser so literal shortcut characters like `[` and `]` still remain typeable when search is focused.
- Chose to split source labels like `radio:` out of queue row titles and into the shared list's leading-prefix field because the user wanted marquee scrolling to leave the source marker visually anchored while only the long track name moves.
- Chose debounced, cancelable queue search over firing a provider request on every keystroke because the user reported that YouTube Music search was too slow and stale requests should be superseded immediately.
- Chose queue-screen-owned async auto-start for the first queued radio stream over starting playback inside the queue service because playback startup can perform remote resolver work and should not block the queue mutation path or freeze Bubble Tea input.
- Chose visible YouTube search-mode chips plus a keyboard mode-cycle binding over keeping the provider unfocused because the user explicitly wanted songs, artists, albums, and playlists to be selectable modes while still being quick to switch from the keyboard.
- Chose one row of source-specific search-kind chips over keeping separate generic filters plus source-specific modes because the user explicitly wanted the queue browser to stop mixing overlapping concepts and shortcuts.
- Chose to treat artist rows as filter selections rather than queueable entries because the user explicitly wanted artist search to narrow later songs results instead of enqueuing non-playable rows.
- Chose grouped queue parent rows for whole album/playlist additions over flattening collection adds into anonymous child tracks because the user explicitly wanted full collections to stay removable as one unit while still allowing child-song removal.
- Chose a square-only root presentation over persistent outer tabs and footer chrome because the user wanted all relevant UI to stay inside one clean centered square at all times.
- Chose full-width row replacement overlays over a more complex transparent layer compositor because Bubble Tea's rendered content path is string-based and the simpler overlay model preserves the art-first layout without introducing fragile terminal-protocol composition logic.
- Chose a muted filler pattern around centered artwork over leaving unused pane space blank because the user wanted the album-art viewport to feel less empty while still making the rendered image bounds visible.
- Chose debounced seek accumulation in the UI over firing one backend seek per keypress because the user explicitly wanted rapid seek taps to collapse into one absolute target while keeping input responsive and the old stream playing during preparation.
- Chose an async playback-artwork lookup with a centered activity overlay over the old synchronous in-render fetch because the user explicitly wanted to see provider attempts and spinner feedback while remote artwork resolution is still in flight.
- Chose an async playback-lyrics lookup over the old synchronous lyrics fetch because real remote resolvers such as `lrclib.net` would otherwise freeze the Bubble Tea event loop whenever the Lyrics pane opened.
- Chose reusable active-line timing in `pkg/lyrics` plus UI-side auto-following over a playback-screen-local timestamp search because karaoke behavior should stay consistent across any future renderer that consumes synced lyrics.
- Chose live visualization repainting over a pane-size-only visualization cache because EQ output must keep moving while the active track plays, even when the viewport never resizes.
